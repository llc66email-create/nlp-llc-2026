StoryWeaver 项目实现框架与逻辑说明（报告版）

一、项目定位与总体目标

本项目是一个面向“哈利波特世界观”互动叙事场景的中文智能系统。它的核心目标不是单纯文本聊天，而是将用户输入转换为“世界内可执行行动”，在保持故事沉浸感的同时持续推进剧情。系统强调三件事：

1) 低延迟交互体验：用户输入后，先快速返回一条短回应（quick），再返回完整叙事（final）。
2) 世界观一致性：通过规则约束、时间线约束、地点与能力约束，减少 OOC（Out Of Character）与设定冲突。
3) 可持续推进：通过任务章节系统、状态管理与RAG检索实现长期可追踪的剧情演化。

从工程形态上看，这是一个“Flask API + 前端页面 + 叙事编排核心 + 可切换生成后端（本地/远程）”的分层架构系统。


二、系统分层架构

项目可以拆分为 6 层：

第 1 层：表现层（Web UI）
- 位置：web_interface/
- 作用：提供角色选择、输入交互、故事展示、状态可视化。
- 重点：main.js 通过 HTTP/SSE 与后端通信，实现 quick + final 的流式呈现。

第 2 层：接口层（Flask API）
- 位置：app.py
- 作用：统一对外接口，负责初始化生命周期、输入校验、请求分流、SSE推送、异常返回。
- 重点：/api/process_input_stream 是体验核心接口；/api/init_status 处理后台初始化可见性。

第 3 层：编排层（StoryWeaver Core）
- 位置：story_weaver/core.py
- 作用：把 NLU、RAG、约束、验证、NLG、状态更新串成一个稳定流程。
- 重点：process_user_input 是主流程入口，决定“先做什么、后做什么、如何回退”。

第 4 层：能力层（NLU / RAG / NLG）
- NLU：意图识别与实体抽取（当前实现主要采用轻量规则策略）。
- RAG：从知识库和交互历史检索相关片段，给生成提供上下文支持。
- NLG：可切换后端（模板、本地模型、Colab远程、API）生成叙事文本。

第 5 层：治理层（约束与验证）
- ConstraintEngine：能力约束、地点约束、时间线约束。
- ValidationPipeline：生成后验证与故事任务推进追踪。
- 作用：将“可生成”提升为“可用、可控、可持续”。

第 6 层：数据与状态层
- GameState：管理角色、地点、物品、剧情节点、交互历史、世界变量。
- Knowledge Base：角色图谱、世界状态、剧情片段、一致性规则。
- 日志与存档：logs/ 下保存交互与会话摘要。


三、启动与初始化逻辑

1) 入口行为
- 用户访问首页时，Flask 不阻塞等待大模型全量加载，而是启动后台线程异步初始化 StoryWeaver。
- 前端通过轮询 /api/init_status 感知系统是否已 ready。

2) 后台初始化内容
- 初始化 NLU 引擎。
- 初始化 RAG Retriever 并加载知识库片段。
- 根据配置初始化 NLG 引擎（远程Colab、本地LoRA、模板或其他后端）。
- 初始化一致性检查器、约束引擎、验证管线、故事进度追踪器。
- 初始化 GameState 并加载世界状态与角色图谱。
- 初始化交互日志系统。

3) 初始化阶段降级策略
- 即使核心系统未完成初始化，接口仍可先返回预定义角色列表，保证前端可先渲染，避免“白屏等待”。
- 若初始化失败，接口明确返回错误信息而不是静默超时。


四、用户交互主链路（端到端时序）

A. 前端发起输入
- 用户在界面输入行动文本。
- 前端优先调用 /api/process_input_stream（SSE）。

B. 后端流式响应两阶段
- 阶段1（quick）：
  - 从当前状态取角色与地点。
  - 结合用户输入与少量RAG背景，构造短提示词。
  - 生成一句短回应，优先追求速度。
  - 进行轻量后验过滤，若命中现代客服/助手话术，替换为世界内兜底句。
  - 通过 SSE event=quick 推送给前端。

- 阶段2（final）：
  - 调用 StoryWeaver.process_user_input 进入完整流水线。
  - 完成意图解析、状态更新、约束注入、RAG检索、叙事生成、后验验证、任务推进、RAG回写。
  - 通过 SSE event=final 推送完整结构化结果。

C. 前端消费结果
- quick 文本先渲染，用户获得即时反馈。
- final 结果到达后更新位置、交互计数、剧情正文、任务完成提示与下一步建议。

D. 兼容回退
- 若流式接口异常，前端自动回退到 /api/process_input 的普通非流式路径，保证可用性。


五、核心模块详细说明

5.1 接口层（app.py）

接口职责可分为四类：

1) 生命周期类
- /：加载首页并触发后台初始化。
- /api/init_status：返回 initializing 或 complete。

2) 游戏准备类
- /api/get_characters：获取角色信息。
- /api/select_character：设置玩家角色和初始位置。
- /api/start_game：返回初始场景和上下文。

3) 交互处理类
- /api/process_input：单次完整处理。
- /api/process_input_stream：流式双阶段处理。

4) 状态与持久化类
- /api/game_status：当前状态快照。
- /api/save_game：保存存档。
- /api/end_session：结束会话并输出总结。

实现重点是“快路径与稳路径并行”：
- 快路径在流式接口 quick 阶段实现，降低首字时延。
- 稳路径由 core.py 完整流程保障一致性与可追踪性。


5.2 编排层（StoryWeaver 核心）

核心思想是“先治理，后生成，再回写”。

完整逻辑顺序如下：

步骤0：现实表达映射为世界内表达
- 对“吃饭、点外卖、看手机”等现实动作做语义映射，转为霍格沃茨语境动作。
- 作用：减少输入端 OOC 直接进入生成模型。

步骤1：意图识别
- 当前以轻量关键词策略为主（move/talk/take）。
- 作用：给后续生成器提供行动方向标签。

步骤2：状态预更新
- 在生成前，根据输入关键词先更新位置（如禁林、魔法部等）。
- 作用：确保生成引用的是“更新后的上下文”，减少叙述错位。

步骤3：构造世界上下文
- 从 GameState 提取当前地点、角色、附近角色、可访问地点等信息。

步骤3.5：约束预检
- 使用 ConstraintEngine 对输入动作做可行性检查（地点规则/时间线等）。
- 不阻断流程，但把违规警告注入后续提示词。

步骤4：RAG检索
- 以用户输入为查询，从知识库取 top_k 相关片段。
- 作用：提供世界事实与剧情先验，减少胡编。

步骤5：一致性检查（当前为简化版）
- 保留扩展位，便于后续强化。

步骤6：NLG生成
- 调用当前配置的 NLG 后端生成叙事结果。
- 支持增强生成器（含动作预测与连贯性支持）与原始生成器两种路径。

步骤6.5：输出后验验证
- 对已生成文本再做约束检查，发现违规则记录告警。
- 作用：形成“输入预检 + 输出后验”的双重防护。

步骤6.6：故事任务推进
- 根据输入与输出文本判断主线/支线任务完成情况。
- 更新章节进度与下一任务提示。

步骤7：反馈回写到RAG
- 将本轮“用户输入 + 系统输出”写回检索库。
- 形成长期记忆闭环，让后续检索可利用历史交互。

最后返回结构化结果：包含 narrative、next_options、state_updates、current_location、story_progress 等。


5.3 NLU模块

当前工程中存在完整 NLU 引擎（意图枚举、实体提取、澄清逻辑），但主流程为保障速度与稳定，采用了更轻量的关键词判定策略。

这体现了一个典型工程取舍：
- 学术能力上，NLU模块具备可扩展深度。
- 工程运行上，主流程优先低时延与可控行为。

因此可以理解为“保留高级NLU能力 + 线上默认轻量执行”的双轨结构。


5.4 RAG模块

RAGRetriever 负责三类工作：

1) 知识库加载
- 启动时读取 plot_segments.json 等内容形成片段池。

2) 检索
- 当前主实现是关键词匹配计分。
- 若无匹配，会随机回退少量片段，避免完全无上下文。

3) 在线增量记忆
- 每轮交互后新增 segment（interaction / generated_narrative），
  使系统具备“用过即记住”的会话增强能力。

注意：
- 文件中保留了 embedding + FAISS 的扩展框架，但当前默认路径为了响应速度，未强依赖该重检索流程。


5.5 NLG模块与实际模型策略

本项目当前实际使用的生成模型是 Qwen2.5-1.5B-Instruct + LoRA 适配器。

部署形态为 ColabRemote：
- 主服务通过 HTTP 调用远程 FastAPI 模型服务。
- 远程服务提供 `/generate` 与 `/two_stage` 两类接口，分别用于快速短回复和双阶段高质量生成。

当前配置为 `USE_COLAB_REMOTE=True`，即优先使用 `COLAB_MODEL_URL` 指向的远程模型服务。

补充说明：
- 系统存在模板降级机制用于故障兜底，但这属于可用性保障路径，不是当前报告中的“实际使用模型”主体。


5.6 约束引擎与验证管线

ConstraintEngine 的价值在于把“世界观合规”从提示词层面前移到规则层面。

约束主要分三类：

1) 能力约束
- 角色在不同时间册别中，哪些咒语可用、哪些不可用。
- 例如某些技能在早期册别中不可出现。

2) 地点约束
- 地点内禁止行为规则。
- 如霍格沃茨内反幻影移形限制、禁林需许可等。

3) 时间约束
- 防止超前知识或事件乱序出现。
- 保证故事在时间线上的合理性。

ValidationPipeline 与 StoryProgressTracker 则负责：
- 把“剧情推进”拆成章节主线+支线任务。
- 根据用户输入与生成文本更新任务状态。
- 输出进度摘要与下一步提示，确保用户有明确推进方向。


5.7 状态管理（GameState）

GameState 维护运行时“世界真相”，包括：
- 角色：位置、状态、属性、关系、背包。
- 地点：描述、在场角色、可交互物品、连接关系。
- 物品：归属与属性。
- 剧情节点：父子关系与标签。
- 交互历史：每轮输入输出与快照。

设计特征：
- 提供快照机制，便于保存与回溯。
- 上下文读取接口明确，方便NLG按状态生成。
- 与日志/存档接口天然兼容。


六、前端交互与体验策略

前端逻辑的关键不在“页面样式”，而在“交互节奏控制”：

1) 初始化可见化
- 前端持续查询 init_status，避免用户误以为系统卡死。

2) 角色先选后玩
- 角色选择成功后才激活输入框，减少无效请求。

3) 流式体验优化
- 先显示 quick，缩短体感等待。
- final 到达后覆盖为完整故事并更新任务信息。

4) 异常可恢复
- 流式失败回退普通接口，用户不需要手工刷新。

这使系统具备“弱网络/弱算力环境下仍可交互”的产品化特征。


七、配置驱动与部署形态

config.py 统一管理模型、数据、系统参数：

- 模型参数：是否用远程、是否用微调、本地路径、超时阈值。
- 数据参数：知识库文件路径、RAG索引目录。
- 系统参数：端口、日志目录、阈值、响应限制。

部署上可分两块：

1) 主服务（本项目）
- Flask 运行在 5001，提供游戏接口与网页。

2) 推理服务（Colab）
- Notebook 启动 FastAPI 暴露 /generate 与 /two_stage。
- cloudflared 提供公网 URL。
- 主服务通过 COLAB_MODEL_URL 调用远程推理。

这种拆分使“业务逻辑”和“重推理负载”分离，便于低成本部署与弹性调优。


八、当前实现中的关键工程策略

1) 双阶段生成策略
- 先快后全：先给一句即时反馈，再给高质量完整叙事。
- 直接提升可用体感，降低大模型冷启动带来的等待焦虑。

2) OOC 风险控制策略
- 输入侧：现实表达映射为魔法世界表达。
- 生成侧：提示词中明确角色、地点、世界约束。
- 输出侧：轻量后验过滤，命中现代助手口吻即兜底。

3) 闭环学习策略
- 将历史交互持续写回RAG，提高后续上下文召回质量。

4) 多后端容灾策略
- 远程模型不可用时，自动降级模板，保证接口可响应。


九、可用于报告的“系统逻辑一句话总结”

该系统本质上是一个“以状态为中枢、以规则为护栏、以检索为补强、以生成为表达”的互动叙事引擎：前端负责节奏，接口负责编排，核心流程负责治理与推进，模型负责文本生成，最终形成兼顾速度、沉浸与一致性的可持续故事交互系统。


十、结论

从实现框架看，这不是“单模型问答应用”，而是一个具备明确工程分层、运行治理机制和产品级容错能力的叙事系统。其关键价值在于：

- 可扩展：模块边界清晰，便于替换NLG或强化RAG。
- 可控：约束与验证机制将世界观一致性前置和后置双重保障。
- 可用：流式双阶段输出与降级回退确保交互连续性。
- 可演化：交互回写RAG与任务推进机制使系统能持续积累上下文能力。

用于报告时，可以将本系统定义为“规则增强的检索-生成一体化互动叙事平台”，并重点强调其在“低延迟体验”和“世界观一致性治理”上的工程实践价值。


十一、实际使用模型与微调方式（仅保留当前线上方案）

本节仅描述当前项目实际运行中使用的模型与微调方式，不包含候选模型、扩展路线或对比方案。


11.1 当前实际使用的生成模型

当前系统实际用于叙事生成的核心模型是：

- 基础模型：Qwen/Qwen2.5-1.5B-Instruct
- 微调形态：LoRA 适配器
- 运行方式：Colab 远程推理（FastAPI）

也就是说，主程序并不是直接在 Flask 进程里做重推理，而是通过 HTTP 请求调用 Colab 侧模型服务：

1) `/generate`：用于快速短文本生成（配合 quick 阶段）。
2) `/two_stage`：用于双阶段高质量生成（hook + full_text）。

主服务通过环境变量 `COLAB_MODEL_URL` 指向该远程推理地址，形成“业务逻辑与推理负载分离”的部署方式。


11.2 当前实际使用的微调方式

当前项目采用的是 LoRA 微调方案，属于参数高效微调（PEFT）路径。其核心思想是：

1) 冻结基础模型大部分参数。
2) 仅训练低秩适配层参数（adapter）。
3) 推理时把 LoRA 适配器挂载到基础模型上。

这种方式在本项目中的实际价值是：

- 训练成本可控：相较全参数微调，显存和训练时长都更可控。
- 版本管理清晰：adapter 文件可以独立管理、替换和回滚。
- 部署兼容性好：Colab 环境下可直接加载基础模型 + adapter 组合。


11.3 当前微调数据组织（实际工程口径）

从项目现有训练数据和运行逻辑看，微调样本重点围绕“哈利波特世界内叙事”构造，核心结构可以归纳为：

1) 用户动作输入（user action）
2) 场景上下文（角色、地点、任务阶段）
3) 约束信息（世界观、时间线、角色能力边界）
4) 目标输出（第一人称、剧情推进、非现代客服口吻）

结合当前系统特性，数据处理侧重点是：

- 强化角色与地点显式注入，减少无上下文泛化回答。
- 清理现代语境话术，降低 OOC 风险。
- 保持输出是“可推进剧情”的叙事，而不是解释型闲聊。


11.4 当前模型在系统中的实际工作分工

Qwen2.5-1.5B-LoRA 在系统中承担的是“文本表达层”的职责，前后由规则与检索共同约束：

1) 输入前：
- Core 层先进行现实动作到魔法语境动作的映射（降低输入侧 OOC）。

2) 生成前：
- 注入角色、地点、RAG上下文与约束上下文。

3) 生成后：
- 由后验过滤与验证流程检查输出，必要时进行兜底。

因此，当前线上效果不是“纯模型能力”，而是“LoRA模型 + RAG补充 + 规则校验”的组合结果。


11.5 当前方案的实际优点与边界

优点：

1) 在现有算力条件下可稳定运行，响应速度与文本质量达到可用平衡。
2) 微调与部署链路已打通，工程维护成本可控。
3) 与约束引擎、任务推进系统协同后，叙事一致性明显优于裸模型直出。

边界：

1) 1.5B 规模在超长复杂剧情上仍可能出现细节漂移。
2) 对高难度隐式约束的遵守仍依赖后验规则补偿。
3) 模型质量提升仍需持续依赖高质量数据迭代。


11.6 报告可直接引用的结论性表述

本项目当前实际采用的模型方案为“Qwen2.5-1.5B-Instruct + LoRA 适配器 + Colab远程推理服务”。在该方案下，系统通过“输入映射、RAG补足、约束校验、双阶段生成”实现了可用的低时延叙事交互。微调方式采用 LoRA 参数高效路径，在不引入全参数训练成本的前提下，显著改善了哈利波特世界观下的文本风格一致性与剧情推进能力。
